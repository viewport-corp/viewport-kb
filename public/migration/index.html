Migration doctrine

Old Docker is reference/prod residue. GitHub is the durable plan. VPS is rebuilt runtime. A live service is not migrated until the repo, env, backup, domain route, smoke test, and rollback all exist.

Why the old model failed

The server became the source of truth. That made completion claims impossible to verify, mixed tenants together, hid automation, and tied business continuity to one fragile host state.

How to avoid repeat

Before any agent changes runtime, it must have an issue, branch, owner, expected files, test command, output artifact, and rollback note. Runtime-only fixes are emergency exceptions.

Root cause board

Why the old structure failed

The failure was not one broken container. It was a control-plane failure: chat, servers, secrets, domains, docs, and product ideas were mixed without durable ownership and verification.

No single control plane

Symptom
Agents claimed completion while state lived in chat, local files, or runtime residue.
Risk
Work could not resume cleanly and proof was scattered.
Doctrine
GitHub issue/branch/PR/evidence file is required for durable work.

Mixed proxy ownership

Symptom
Client, ops, tenant, and experiment routes shared the same live VPS/proxy surface.
Risk
One route fix could break another business.
Doctrine
Every route needs domain group, repo, service, owner, proof.

Bot identity conflicts

Symptom
ViewportBot, Hermes, OpenClaw, Codex, and Claude all acted as operators without one authority map.
Risk
Duplicate work, false confidence, and unclear responsibility.
Doctrine
Hermes is the operator shell; other agents are bounded workers.

Secrets in human channels

Symptom
Passwords, tokens, auth paths, device codes, and PAT references appeared in chats/reports.
Risk
Credential takeover and polluted evidence history.
Doctrine
Rotate, store in secret manager/env only, never publish values.

Foreground execution loops

Symptom
Long fixes ran through Telegram messages and timed out or lost state.
Risk
Production work died when chat runtime failed.
Doctrine
Long jobs run in background with logs, PRs, runbooks, and resume files.

Tenant boundary drift

Symptom
Corporate, partner, associate, client, and personal work mixed in the same runtime surface.
Risk
Wrong files, wrong domains, wrong permissions, wrong client exposure.
Doctrine
Tenant repos and namespaces isolate private client/partner work.
