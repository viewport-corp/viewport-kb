1. DIRECT VERDICT

Yes, with a critical correction to the framing.

The Obsidian→Hermes→NotebookLM→publish→repeat loop Sam described is the right shape. A closed loop where input is captured, classified, executed by an agent, verified, and published back is the correct pattern for a company this size running on AI agents. Viewport should absolutely build this.

The correction: that loop does not exist at Viewport today, and building the full version before v0.1 runs once will produce the same result as the last 30-40 days. 488 Hermes sessions, 29,009 messages, and zero completed GitHub issue cycles is exactly what happens when you build the top of the stack before the bottom works.

The verdict is: build the loop. Build it bottom-up. Ship nothing at the top until the foundation closes once.

The five prerequisites that must all exist simultaneously before any automation is real:

1. Durable task intake — Sam's instruction converts to a machine-readable task any agent can pick up

2. Shared context — agent knows current state without re-deriving it

3. Authority grant — agent is permitted to act on production

4. Verification loop — agent completes work, attaches evidence, closes the task

5. Memory writeback — what was done persists so the next agent doesn't repeat it

Zero of five exist right now. This report shows how to build all five in the right order.

2. THE EXACT VIEWPORT LOOP

Nine stages, with Viewport-specific intake, agents, and outputs named explicitly.

CAPTURE
  Surfaces: Telegram → Hermes | Slack #ops | GitHub issue comment | Sam voice note
  Output: raw instruction record with source, timestamp, session_id
  Owner: Hermes (primary), Claude Code (fallback for session-bound work)
        ↓
CLASSIFY
  Input: raw instruction record
  Rules: tenant (viewport|mlg|mlh|bccl|saathi) × domain (runtime|dns|secret|docs|code|infra|finance)
         × priority (P0|P1|P2) × agent-seat (hermes|openclaw-<role>|claude|codex|human-sam)
         × required_approval (yes|no) based on authority gateway policy
  Output: labeled GitHub issue in viewport-corp/viewport-ops with task-packet schema fields
  Owner: Hermes
        ↓
STORE
  Input: classified GitHub issue
  Action: write to GitHub issue body (source of truth) + append to tracker.json + optional KB entry
  Rule: if tenant ≠ viewport, also write tenant-scoped stub to correct tenant repo
  Owner: Hermes
        ↓
RETRIEVE
  Input: assigned agent wakes from cron or event trigger
  Action: read STATE.md → active issue body and last 5 comments → tracker.json last 10 → weekly digest
  Rule: if any source inaccessible → write BLOCK comment on issue, alert Sam via Telegram
  Output: full context packet in agent working memory before first tool call
  Owner: assigned agent (OpenClaw seat, Claude Code, Codex)
        ↓
DECIDE
  Input: full context packet + authority gateway policy
  Rule: if action is in pre-authorized set → proceed | if action requires approval → write decision
        packet to issue, post Telegram alert, wait for Sam label `approval:granted`
  Outputs: action plan attached to issue as comment; lease record written
  Owner: assigned agent + authority gateway check
        ↓
EXECUTE
  Input: approved action plan + active lease
  Rule: one lease per repo/domain/tenant combination at a time (lease table enforced)
  Outputs: commands run, commits pushed, PRs opened, service changes applied
  Owner: assigned agent within lease boundary
        ↓
VERIFY
  Input: completed execution artifacts
  Action: run defined acceptance criteria checks (CI, health endpoint, git log, diff)
  Rule: if verification fails → write FAIL comment, do NOT close issue, alert Sam
  Output: verification result (PASS|FAIL) attached to issue
  Owner: verifier (separate from executor — can be Hermes reading execution output)
        ↓
PUBLISH
  Input: verified artifacts
  Targets: GitHub issue (close with evidence links) | tracker.json append | viewport-kb/decisions/
            | public /migration/reports page if public-safe | status.json update | Telegram summary
  Rule: private evidence goes to private evidence store only; public pages get redacted export
  Owner: Hermes
        ↓
LEARN
  Input: closed issue + evidence bundle + verification result
  Action: write decision summary to viewport-kb, update agent seat's workspace context,
          update STATE.md `last_completed`, append tracker.json event, check drift vs STATE.md
  Rule: no new session starts without reading the last LEARN output for its domain
  Owner: Hermes + any agent that will pick up next task in same domain

Critical invariant: The loop only closes when the GitHub issue closes with a linked evidence bundle. A session that ends without closing the issue is not a completed loop — it is a partial run that must be resumed, not restarted.

10. HOW CLAUDE / CODEX / OPENCLAW / HERMES INTERACT WITHOUT SPLIT-BRAIN

The split-brain problem today: three separate memory stores (Hermes state.db, OpenClaw SQLite, Claude Code session) with no bridge. Same facts re-derived 488 times. Same blockers rediscovered. This section specifies the protocol that prevents it.

---

The Single Rule

GitHub is the shared brain. All agent coordination goes through GitHub issues and tracker.json. No agent-to-agent direct communication in v0.1.

No Slack channels between agents. No direct API calls from Hermes to OpenClaw. No shared SQLite. No shared filesystem beyond read-only access to the same repos. Every coordination message is a GitHub issue comment.

---

Session Start Protocol (every agent, every session)

Before any agent begins work on anything, it reads in this exact order:

1. Migration/council/STATE.md — current phase, active round, Sam's last answer, automation gate status

2. The single state:active GitHub issue — body and last 10 comments

3. tracker.json — last 10 events

4. viewport-kb/digests/ — most recent weekly digest

5. authority-gateway/policy.yaml — what this agent is allowed to do

If any of steps 1–3 is inaccessible (mount missing, auth failure, file not found), the agent writes a state:blocked comment on the active issue and sends a Telegram alert before doing anything else.

---

Task Routing

Sam → Hermes → GitHub issue: the only routing path in v0.1.

Sam says "work on X" in Telegram. Hermes creates or updates a GitHub issue with task-packet schema. Hermes adds the agent label (agent:openclaw, agent:claude, etc.). The assigned agent's cron runs, finds issues with its label in state:active, and picks up work. If multiple issues have the agent's label, it picks the highest priority one.

No Hermes→OpenClaw direct call. No OpenClaw→Hermes callback. Only GitHub.

---

Handoff Protocol

When Agent A finishes and Agent B must continue:

1. Agent A writes evidence bundle to viewport-company-os/evidence/

2. Agent A comments on the issue with: HANDOFF TO: <agent-b> | Completed: <what was done> | Next: <exact action for agent-b> | Evidence: <path>

3. Agent A removes its label, adds agent-b's label

4. Agent A appends tracker.json event with event_type: handoff

5. Agent B's cron picks up the issue on next run

No shared memory writeback needed. The issue comment IS the handoff.

---

Conflict Prevention (Lease Enforcement)

Before any mutation, the executing agent:

1. Reads leases/active-leases.json from GitHub

2. Checks for active lease on same tenant + domain

3. If conflict: writes a blocked comment and stops

4. If clear: writes its own lease entry, commits to GitHub, then begins work

5. After work completes: marks its lease state: released, commits

Race condition note: Two agents reading simultaneously could both see no conflict. In v0.1, this is acceptable because only Hermes and Claude Code are likely to run simultaneously, and they work in different domains. A lease commit conflict (git merge conflict on the lease file) is the detection mechanism — whichever agent's commit is rejected must back off.

---

Memory Writeback

At session end, any agent that completed meaningful work writes:

• A decision entry to viewport-kb/decisions/ if an architectural choice was made

• An event to tracker.json

• A comment on the closed issue with evidence bundle path

Nothing else. No private SQLite sync. No session DB export. The agent's private session memory is disposable. What goes into GitHub survives.

12. RISKS AND SAFETY GATES

Risks are ordered by current likelihood of causing catastrophic unrecoverable loss.

---

Risk 1: PAT with admin:enterprise scope used during recovery work

Likelihood: High. It is the current active token.

Impact: Any session using this PAT has permissions to delete repos, remove org members, modify org settings. If the token is logged (it is — ghp_ pattern appears 138 times in state.db), any process that reads state.db has these permissions implicitly.

Gate: TP-01 must complete before any automation uses GitHub credentials. Until TP-01 closes, all GitHub operations require Sam to authenticate in the browser. No automated git push using the current broad PAT.

---

Risk 2: Loop appears to close but evidence is not in GitHub

Likelihood: High. This is exactly what produced p0_foundation_completion.status: completed_safe_foundation with the artifacts on an unpushed branch.

Impact: False progress signal. Sam reads status as green, starts new work, discovers broken foundation 3 days later.

Gate: "Done" means: evidence YAML is committed to main, GitHub issue is closed, tracker.json is appended, git log --oneline origin/<branch> confirms push. Any claim of completion without all four checks fails the acceptance criteria automatically.

---

Risk 3: Hermes restart breaks Telegram intake

Likelihood: Medium. Mount requires gateway restart. Prior audit found gateway restart was blocked specifically because of this risk.

Impact: Hermes goes offline. Sam loses his primary agent interface. Recovery requires manual VPS access and docker-compose debugging.

Gate (TP-07): Sam must explicitly approve restart in writing ("I approve Hermes restart for migration mount"). Hermes tests Telegram connectivity within 60 seconds of restart. If no Telegram self-test message received in 90 seconds, Hermes automatically rolls back the docker-compose change and alerts via alternative channel (email or GitHub issue comment).

---

Risk 4: Duplicate task pickup — two agents work the same issue simultaneously

Likelihood: Medium, once multiple agents are running with crons.

Impact: Conflicting commits, double-spend of work, potential conflicting evidence bundles.

Gate: Lease system (Section 3.3). Any agent that cannot acquire the lease on first GitHub commit must back off and post a blocked comment. The lease conflict is detected at git push time — a merge conflict on active-leases.json means the race was lost.

---

Risk 5: Morning brief cron sends private data to Telegram

Likelihood: Medium. Hermes has access to STATE.md, private session context, and evidence paths.

Gate: Morning brief template is constrained to: issue number, title, current state label, last tracker.json event type, and a one-line "next action" field. No raw content, no credential patterns, no paths to private evidence. The cron job has a pre-send sanitizer that rejects output containing sk-, ghp_, xoxb-, AIza, passwords, or email addresses.

---

Risk 6: Scope creep before first loop closes

Likelihood: High. The 30-40 day session evidence shows four recurring instances of this exact pattern.

Impact: The 48-hour build plan (Section 4) becomes a 7-day build plan, then a 14-day build plan, and v0.1 never actually closes.

Gate: This report's Stop list (Section 5) is the gate. Sam reads it and says "stops confirmed" in Telegram before Hermes begins Day 1. Hermes treats any instruction that appears on the stop list during the 48-hour window as out-of-scope and responds with: "That item is on the v0.1 stop list. I'll add it to the v0.2 backlog issue and continue with the current task."

---

Risk 7: Active issue gets manually labeled state:done without evidence

Likelihood: Low but catastrophic. One label click produces a false completion signal that could trigger downstream automation in v0.2.

Gate: CI check on any issue receiving state:done label that has no committed evidence bundle path in its body. CI fails and adds a comment: "Missing evidence bundle. Add evidence_bundle_path: field before closing." Sam can override by adding approval:granted label, but the override is logged.

How /migration/loop Gets Updated

1. Hermes runs scripts/update_loop_data.py — reads closed issue metadata and evidence bundle

5. /migration/loop serves new data within 60 seconds

The loop page is only ever as current as the last closed issue. If no issue closes for 48 hours, the page goes yellow. If no issue closes for 7 days, the page goes red. This creates a visible accountability signal that does not require Sam to check GitHub directly.
