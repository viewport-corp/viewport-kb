ADDENDUM: Why Viewport Is Not Automated

Date: 2026-06-08 | Auditor: Claude Sonnet 4.6

Trigger: Sam's voice correction — Viewport is intended as a trillion-dollar AI company operating every country, every industry, every language, every department and product with centralized automated agents. Why is it not automated today?

This addendum does not repeat the original report. It answers Sam's question directly, corrects one framing error in the original report, and uses the 30-40 day session evidence to show the specific loop patterns that caused the failure.

---

THE DIRECT ANSWER

Viewport is not automated because the prerequisites for automation were never assembled in the right order. The vision is fully documented and directionally sound. The infrastructure to execute that vision — task intake, shared memory, authority grant, and always-on execution — does not exist as a closed system. Every session over 30-40 days built pieces of the top of the stack while leaving the bottom unfinished.

For Sam to say "work on this" and have centralized agents execute, five things must all exist simultaneously:

PrerequisiteStatusGap
---------
Durable task intake — Sam's instruction converts to a machine-readable task that any agent can pick upNot closedTelegram → Hermes is one session. No queue. No persistent pickup.
Shared context — agent knows the current state without re-deriving it from scratchNot closedThree separate memory stores, no bridge, each session starts cold
Authority grant — agent is permitted to act on production systemsHard-blockedautomation_gate: No expanded production autonomy until secrets scoped/rotated never unblocked
Verification loop — agent completes work, posts evidence, closes the taskNever completed once139 tasks, 1 done
Memory writeback — what was done persists so the next agent doesn't repeat itNot closedSession ends, files on VPS or unpushed branch, next session rediscovers same state

All five must be working before "work on this" becomes "agents execute." Right now, zero of five are closed.

---

WHY SAM CAN'T JUST SAY "WORK ON THIS"

The chain from Sam's instruction to agent execution breaks at every joint:

Sam says "work on this" in Telegram
    ↓
Hermes receives the Telegram message — JOINT 1: WORKS
    ↓
Hermes reads shared context to understand current state — JOINT 2: BROKEN
    (Hermes has no access to /opt/data/migration — mount missing since day one)
    (STATE.md is on a branch 6 commits ahead of remote, never pushed)
    (Hermes reads session memory only — 488 sessions, each starting cold)
    ↓
Hermes routes task to the right specialist agent — JOINT 3: BROKEN
    (No shared task queue — Hermes uses Telegram, OpenClaw would use GitHub issues)
    (No routing protocol — task-lease.schema.yaml exists as a schema, not an enforcer)
    (No message bus between Hermes and OpenClaw)
    ↓
Specialist agent executes with production permissions — JOINT 4: HARD BLOCKED
    (automation_gate = no expanded autonomy until secrets rotated)
    (OpenAI sk- 970 hits, ghp_ 138, Telegram tokens 276 in session DB)
    (PAT scope is admin:enterprise, expiry unknown, leaked predecessor never revoked)
    ↓
Agent writes evidence and closes the task — JOINT 5: BROKEN
    (No agent writes to a shared store — each writes to its own SQLite/markdown/state.db)
    (Council tracker.json never appended to after bootstrap — 29 days of empty log)
    (GitHub issue loop closed zero times)
    ↓
System knows what was done for the next session — JOINT 6: BROKEN
    (No memory writeback from completed tasks)
    (Next session starts with same blockers as previous session)

The intake joint works. Every other joint is broken. This is why Sam can give an instruction and Hermes can respond intelligently — but nothing actually gets finished and the next day starts from zero.

WHY HERMES AND OPENCLAW ARE NOT CENTRALIZED

Centralization requires a single intake point that all agents read from. That does not exist.

Hermes reads from Telegram. OpenClaw reads from nothing (0 running crons, no task intake). Claude Code reads from the current session. There is no shared queue.

For "work on this" to execute centrally:

1. Sam posts instruction to Telegram (works)

2. Hermes converts it to a GitHub issue in the same session (never implemented — Hermes creates sessions, not issues)

3. GitHub issue is labeled with target agent seat (label → routing — schema exists, not enforced at runtime)

4. Agent with that seat wakes up via cron, reads the issue, executes (0 running crons)

5. Agent posts evidence as issue comment and closes it (closure loop — closed zero times)

6. Morning brief summarizes what closed (cron not running)

Instead: Sam posts to Telegram → Hermes session reads it → Hermes works in session → session ends → nothing in GitHub → no agent picks up anything → next session starts from zero.

Centralization requires shared memory. Three separate stores exist with no bridge.

• Hermes state.db (/opt/data/state.db): 488 sessions, 29k messages, SQLite. Contains credential patterns. Not readable by OpenClaw. Not readable by Claude Code. Hermes itself cannot currently read the migration evidence (/opt/data/migration not mounted).

• Fresh OpenClaw: 26 workspace agent folders, SQLite memory per agent. No connection to Hermes state.db. No writing back to GitHub.

• Claude Code: reads files from VPS and GitHub. No connection to either other store.

The centralized-agent-registry.yaml file — listed as a P0 foundation artifact — lives on a local branch that was never pushed. No agent can read it. The symbol of centralization is itself not centralized.

Centralization requires an authority grant. This was hard-blocked for the entire 30-40 day period.

The automation gate (No expanded production autonomy until categories are scoped/rotated/stored safely) was never unblocked because the secrets rotation that would unblock it was never completed. This means: even if the task queue and shared memory existed, agents were explicitly prevented from taking production actions. The gate posture is correct. But it became the permanent operating state instead of a temporary blocker.

CORRECTION TO THE ORIGINAL REPORT

The original report concludes: *"The architecture is right. The foundation is real. The only thing missing is one closed loop that proves it works."*

This framing is accurate but undersells the gap relative to Sam's vision. One closed loop proves the mechanism. But Viewport's goal is not one closed loop — it is automated centralized agents operating every country, industry, language, department, and product.

The corrected framing:

One Closed Loop (v0.1)Trillion-Dollar Automation
------
Sam tells Hermes what to do each sessionSam sets goals; agents break down work and assign it autonomously
Hermes picks up tasks from TelegramAll agents pick up tasks from a shared queue, no human routing
Evidence written by whichever agent did the workEvidence written to a shared canonical store all agents can read
Daily morning brief summarizes what happenedContinuous event log; agents self-correct without daily summary prompts
Human approval for each production actionAgents with scoped authority act without approval on pre-authorized task types
One issue, one agent, one sessionN issues, M agents, parallel sessions, lease system, deduplication

The 7-day recovery plan is the correct first step. Sam should be clear-eyed that completing it does not mean agents will autonomously execute everything. It means the mechanism has been demonstrated to work once. Full automation — the trillion-dollar model — additionally requires: always-on agent fleet with event-driven task pickup (not session-based), scoped authority model with pre-authorized task categories, shared memory writeback, drift detection and self-healing watchers, and multi-tenant runtime isolation.

None of those are wrong ideas. They are the right ideas. They are dependent on v0.1 existing first.

---

THE GAP IN ONE SENTENCE

Viewport has a company OS for a trillion-dollar automation company — written in YAML, with 26 agent seats, documented across 163 files — but no running cron, no shared task queue, no closed authority loop, and no completed GitHub issue cycle, because the prerequisites for automation were built out of order: the company OS was built before the control plane was bootstrapped, and the control plane has been stuck on pat_revoked: false since 2026-05-10.
